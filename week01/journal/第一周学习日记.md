## 撤销变更

在 Git 里撤销变更的方法很多。和提交一样，撤销变更由==底层部分（暂存区的独立文件或者片段）==和==上层部分（变更到底是通过哪种方式被撤销的）==组成。我们这个应用主要关注的是后者。

主要有两种方法用来撤销变更 —— 一是 `git reset`，还有就是 `git revert`。接下来咱们逐个进行讲解。

1. <b>git reset</b>[仅建议在本地仓库使用]

   回退之后，提交历史消失，但是工作区c2做的修改还在

   虽然在你的本地分支中使用 `git reset` 很方便，但是这种“改写历史”的方法对大家一起使用的远程分支是无效的哦！

   ### 为什么不能这样做？

   #### 1. **历史分叉**（Diverged History）

   ```
   # 远程仓库的历史
   A---B---C  (origin/main)
   
   # 你reset后的本地历史
   A---B  (main)
   
   # 当你试图推送时，Git发现：
   # - 远程有C提交
   # - 你没有C提交
   # Git会拒绝，因为这样会丢失C提交
   ```

   

   #### 2. **强制推送的危险**

   ```
   # 你可以强制推送
   git push --force origin main
   
   # 但这样会：
   # - 删除远程的C提交
   # - 小明下次pull时会报错
   # - 小明的本地历史和你不同步
   ```

2. <b>git revert</b>[可以在远程分支使用]

​	相当于再次提交一个撤销了c2更改的新提交

​	简单说：**已公开的commit就像泼出去的水，收不回来，只能"道歉"（revert）**。

## 1.git是怎么工作的？[ai回答留档]

​	将你的某个文件夹看作本地仓库，在文件夹内通过远程与在另一个服务器上的仓库连接

​	git操作分为工作区，暂存区（本地），远程仓库

| 工作区域                  | 概念解释                                                     | 状态与操作                                                   |
| :------------------------ | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **工作区 (Workspace)**    | 就是你电脑上能看到的项目文件夹。你在这里编写代码、添加文件，进行所有日常的创作和修改。 | 文件初始状态为**未跟踪 (Untracked)** 或**未修改 (Unmodified)**。当你修改文件后，状态变为**已修改 (Modified)**。 |
| **暂存区 (Stage/Index)**  | 可以理解为一个“提交预演区”或“购物车”。它让你能精心挑选想要记录到历史中的修改。你可以把多个文件的修改先放进去，最后打包成一个有意义的版本（比如“修复了登录页面的样式错误”）。 | 主要操作是 `git add`，这个命令会把工作区中选中的修改“添加”到暂存区，文件状态变为**已暂存 (Staged)**。 |
| **本地仓库 (Local Repo)** | 这是 Git 的核心，位于项目文件夹下的隐藏目录 `.git` 中。它存储了项目所有的历史版本和元数据，就像你私人时光机的“记忆核心”。 | 主要操作是 `git commit`，这个命令会把暂存区中的所有修改，作为一个新的“版本”永久地记录到本地仓库中。之后，文件状态又恢复为**未修改 (Unmodified)**。 |

## 2.git的本地仓库核心流程

​	**本地仓库的核心流程**可以用这个简单的命令流来概括：

1. **开发**：在工作区修改代码 (`git status` 查看状态)。
2. **暂存**：`git add <文件名>` 或 `git add .` (添加所有改动)。
3. **提交**：`git commit -m "你的提交说明"`，将暂存区的改动打包成一个新版本存入本地仓库。
4. **查看历史**：`git log` 查看所有提交记录。

## 3.git远程仓库相关

​	当你想和别人协作，或者想在另一台电脑上继续你的工作时，就需要一个“远程仓库”。它是一个存放在**服务	器上**的仓库副本，可以被团队成员共同访问。

​	你可以把远程仓库想象成一个**团队的公共公告板**，所有人都可以把本地的工作成果“贴”上去，也可以从上面“	取”下别人贴的最新内容。知名的远程仓库托管平台有 **GitHub**、**Gitee**、**GitLab** 等。

​	远程仓库和本地仓库的协作通常是这样进行的：

1. **克隆 (Clone)**：当你加入一个新项目，第一步通常是把服务器上的远程仓库**完整地复制**到你的本地电脑上。这个命令会自动在你的电脑上创建一个本地仓库，并将远程仓库设置为它的“搭档”。

   bash

   ```
   git clone <远程仓库的地址>
   ```

   

2. **推送 (Push)**：当你在本地完成了一些工作，并提交到了本地仓库后，你想把这些成果分享给团队。这时，你需要执行 **`git push`** 命令，把你本地仓库的特定版本“推送”到远程仓库上。

   bash

   ```
   git push origin master
   ```

   

   *(这里 `origin` 是远程仓库的默认名称，`master` 是你想推送的分支名)*

3. **拉取 (Pull)**：在开始一天的工作前，或者你想看到同事的最新进展，你需要执行 **`git pull`** 命令，从远程仓库把别人推送的最新版本“拉取”到你的本地仓库，并自动合并到你当前的工作中。

   bash

   ```
   git pull origin master
   ```

## 一些学习过程中的疑问

​	**1.本地仓库本质上就是你的某个文件夹吗**

​	兑＜（＾－＾）＞



​	**2.怎么连接远程仓库，是在一个文件夹内进行了连接远程仓库之后这个文件夹就不用再次连接了吗**



### 问题一：怎么连接远程仓库？

连接远程仓库，本质上是告诉你的本地 Git：“嘿，在某个服务器地址上，有一个仓库是你的‘搭档’。” 主要有两种场景：

#### 场景1：你还没有本地仓库，想从零开始

这是最常见的“先把远程仓库拉下来”的情况，比如你在GitHub上创建了一个新项目。

- **操作**：使用 **`git clone`** 命令。

  bash

  ```
  git clone https://github.com/你的用户名/你的仓库名.git
  ```

  

- **效果**：这个命令会做三件事：

  1. 在你当前目录下创建一个和仓库同名的文件夹。
  2. 把这个文件夹初始化为一个Git本地仓库。
  3. 自动将你克隆的远程仓库地址，设置为这个本地仓库的“搭档”，并默认给它起名为 **`origin`**。

- **结论**：这是最省心的方法，一步到位，连接也自动完成了。

#### 场景2：你已有一个本地仓库，想把它关联到一个远程仓库

这种情况通常是你先在本地开始了项目（比如用 `git init` 创建了仓库），之后才决定把它放到远程（如GitHub）上去管理。

- **操作**：分两步走。

  1. 先在远程（如GitHub）上**创建一个空的新仓库**（创建时不要勾选“初始化README”）。
  2. 然后在本地仓库中，执行 **`git remote add`** 命令，手动指定远程仓库的地址。

  bash

  ```
  # git remote add <远程仓库的本地称呼> <远程仓库的地址>
  git remote add origin https://github.com/你的用户名/你的仓库名.git
  ```

  

- **解释**：

  - `remote add`：意思是“添加一个远程仓库的关联”。
  - `origin`：这是远程仓库在你本地的**别名**。因为真实的服务器地址通常又长又难记，所以Git允许你用一个简短的名字代替它。`origin` 是约定俗成的默认名称，代表“主远程仓库”。
  - 后面的URL就是远程仓库的真实地址。

### 问题二：连接一次之后，这个文件夹就不用再次连接了吗？

**是的，完全正确！**

一旦你在一个文件夹（即本地仓库）内执行了 `git clone` 或 `git remote add`，这个连接信息就会被永久地记录在这个文件夹的 **`.git` 隐藏目录**中。

以后你在这个文件夹里工作时：

- **只需要使用 `git push`**：Git会自动知道要推送到哪个远程仓库（默认是 `origin`）的哪个分支（比如 `main` 或 `master`）。
- **只需要使用 `git pull`**：Git会自动知道从哪个远程仓库拉取最新的代码。

除非远程仓库的地址发生了变化（比如服务器迁移），否则你**不需要、也不应该**再次执行连接命令。

### 🗺️ 用一个类比帮你彻底理解

你可以把这个过程想象成**给你的手机（本地仓库）设置一个家庭 Wi-Fi（远程仓库）**：

1. **`git clone` （场景1）**：
   - 就像你买了一部新手机（本地仓库），店员已经帮你把家里的 Wi-Fi 密码和名称（远程仓库地址和别名 `origin`）都设置好了。你到家就能直接上网（`push`/`pull`）。
2. **`git remote add` （场景2）**：
   - 就像你已经有了一部手机（已存在的本地仓库），然后搬了新家。你需要手动打开设置，输入新家的 Wi-Fi 名称和密码（执行 `git remote add origin <地址>`），从此手机就记住这个新家了。
3. **“不用再次连接”**：
   - 就像你的手机一旦成功连接上家里的 Wi-Fi，以后只要你在家，手机就会自动连接。你不需要每次上网前都去输入一遍密码。只要你不“忘记这个网络”（即不删除 `.git` 文件夹或其中的配置），这个连接就一直有效。

总结与验证

- **如何连接？**

  - 新项目用 `git clone`（最常用，最简单）。
  - 已有本地项目用 `git remote add origin <地址>`。

- **如何验证连接成功？**
  在你项目的根目录下，可以随时运行这个命令来查看当前配置的远程仓库信息：

  bash

  ```
  git remote -v
  ```

  

  它会显示你本地仓库关联的所有远程仓库的别名和地址。如果看到了类似下面的内容，就说明连接成功了：

  text

  ```
  origin  https://github.com/你的用户名/你的仓库名.git (fetch)
  origin  https://github.com/你的用户名/你的仓库名.git (push)
  ```

## 为什么 `println` 会自动调用你写的 `toString`？

### 1. 秘密藏在 `println` 的源代码里

当你调用 `System.out.println(user)` 时，`println` 方法其实有很多个重载版本。对于对象来说，它会调用这个版本：

java

```
// 这是 Java 标准库中 PrintStream 类的源码（简化版）
public void println(Object x) {
    String s = String.valueOf(x);  // 关键在这行！
    // ... 然后打印字符串 s
}
```



### 2. `String.valueOf()` 又调用了什么？

接着看 `String.valueOf()` 的源码：

java

```
// 这是 Java 标准库中 String 类的源码（简化版）
public static String valueOf(Object obj) {
    return (obj == null) ? "null" : obj.toString();  // 关键在这行！
}
```



看到没？**`String.valueOf()` 内部调用了参数的 `toString()` 方法！**

### 3. 多态机制起作用了

这里发生了两件事：

java

```
User user = new User(1001, "zhangsan", "STUDENT", null, "101");

// 当你执行：
System.out.println(user);

// 实际调用链是：
// 1. println(Object x) 被调用，参数 user 被当作 Object 类型看待
// 2. println 内部调用 String.valueOf(user)
// 3. String.valueOf 内部调用 user.toString()
// 4. 关键：由于多态，虽然 user 被当作 Object 类型，但实际执行的是 User 类的 toString()！
```

## 什么时候必须加 `this`？

### 情况1：参数名和成员变量同名（最常见）

java

```
public class Status {
    private final String code;  // 成员变量
    
    // 构造方法中：参数名也叫 code
    public Status(String code, String description) {
        this.code = code;        // 必须加 this！区分成员变量和参数
        this.description = description;
    }
    
    public String getCode() {
        return code;  // 这里没有同名参数，可以不加
    }
}
```



在构造方法中，如果不加 `this`：

java

```
// 错误示范
public Status(String code, String description) {
    code = code;  // ❌ 自己给自己赋值，成员变量没变！
    // 这里的 code 都是指参数 code
}
```

### 情况2：局部变量隐藏了成员变量

java

```
public void someMethod() {
    String code = "local";  // 局部变量
    System.out.println(code);      // 输出 "local"（局部变量）
    System.out.println(this.code); // 输出成员变量的值
}
```
